## 2. Scope Delivered

### 2.1 EcsEngine.Core additions

- **`IWorldOperation.cs`** — `IWorldOperation { void Apply(in ReplayContext ctx); }` marker interface for all recorded mutations.  `ReplayContext` bundles the target `EcsWorld` and optional `TickScheduler?` into a single `readonly record struct`, eliminating the original two-method interface.
- **`OpLog.cs`** — in-memory ordered log of `IWorldOperation` entries; attached to an `EcsWorld` via `AttachOpLog(OpLog)`.  Exposes `IReadOnlyList<IWorldOperation> Operations` and the run `Seed`.
- **`TickScheduler.cs`** — adaptive tick interval manager.  `SetInterval(int ms)` updates the interval and records a `SetTickIntervalOperation` to any attached op-log.  `SetIntervalDirect(int ms)` (internal) sets the interval without recording, used by `WorldReplayer` during replay.
- **`SnapshotMode.cs`** — `TickBoundary` (no pending mutations allowed) and `Immediate` (mid-tick, best-effort) variants.
- **`Replay/WorldOperations.cs`** — eight concrete `IWorldOperation` record types:
  - `CreateEntityOperation`, `MarkForDeletionOperation`
  - `AddComponentOperation<T>`, `RemoveComponentOperation<T>`
  - `QueueEventOperation<T>`, `AdvanceTickOperation`
  - `ApplySafePointOperation`, `SetTickIntervalOperation`
- **`EcsWorld.cs`** — recording plumbing: `AttachOpLog(OpLog)`, `DetachOpLog()`, `AliveEntityIds`, `CreateEntityWithId(int)`, `SetTick(int)`.  Every public mutating method (`CreateEntity`, `MarkForDeletion`, `QueueAddComponent`, `QueueRemoveComponent`, `QueueEvent`, `AdvanceTick`, `ApplySafePoint`) records to the attached op-log when one is set.
- **`InternalsVisible.cs`** — `[InternalsVisibleTo]` grants for `EcsEngine.Replay` and `EcsEngine.Replay.Tests`.

### 2.2 EcsEngine.Replay library (new project)

- **`IComponentSerializer.cs`** — per-type contract: `HasComponent`, `WriteComponent`, `ReadAndApply`.
- **`ComponentSerializerRegistry.cs`** — `Register<T>(write, read)` stores typed delegates; serializers iterated sorted by `Type.FullName` for determinism (D-057).  `TryGetSerializer` guarded by `Debug.Assert`.
- **`ComponentHasherRegistry.cs`** — `Register<T>(Func<T, uint>)` stores per-type hash lambdas sorted by type name.  `Register<T>()` no-arg overload delegates to `DeterministicHasher<T>.Default` (added in M6).
- **`WorldHasher.cs`** — FNV-1a 64-bit hash over: world tick, alive entity IDs sorted ascending, then per registered component type the `(entityId, componentHash)` pairs sorted by entity ID (D-058).
- **`WorldReplayer.cs`** — creates a fresh `EcsWorld`, iterates `OpLog.Operations`, calls `op.Apply(in ctx)` with a `ReplayContext` containing the target world and optional `TickScheduler`.
- **`SnapshotWriter.cs`** — binary format (D-056): 4-byte magic `ECSS`, 2-byte version, 1-byte `SnapshotMode`, 4-byte tick, sorted entity list, per-type serialized component blobs.  Enforces no-pending-mutations invariant under `TickBoundary`.
- **`SnapshotReader.cs`** — validates magic/version, restores entities and components, applies all component adds via a single `ApplySafePoint`.  Throws `InvalidDataException` on unknown type names or corrupt headers.

### 4.3 AOT / Reflection Validation
- `ComponentSerializerRegistry` and `ComponentHasherRegistry` use generic delegates — no reflection on any hot path.
- `WorldHasher.Hash` is fully reflection-free; iterates pre-registered lambdas over `QueryEach<T>`.
- `SnapshotWriter`/`SnapshotReader` use `Type.FullName` only for serialization key lookup, which is safe under trim/AOT.
- `WorldReplayer.Run` iterates an `IReadOnlyList<IWorldOperation>` with virtual dispatch — no reflection.

## 5. Performance and Allocation Summary
- `WorldReplayer.Run` allocates one `EcsWorld` and invokes one virtual dispatch per recorded operation.  No additional heap allocation per operation beyond what the operations themselves require.
- `WorldHasher.Hash` allocates one sorted `int[]` of entity IDs and one `List<(int, uint)>` per registered component type per call; acceptable for a diagnostic/testing tool, not intended as a per-tick hot path.
- `SnapshotWriter.Write` allocates a sorted entity array and iterates serializers — one-time I/O cost.
- No benchmarks added for M5; none of the new APIs are on the primary simulation hot path.

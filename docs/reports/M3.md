## 5. Performance and Allocation Summary
- SoA chunk layout ensures sequential memory access during `QueryEach`; effective cache utilisation at all entity counts.
- Constant 120 B allocation per `QueryEach` call confirmed by `[MemoryDiagnoser]`. All iterations themselves are allocation-free.
- `QueryRegistry` cache eliminates archetype scanning after first call per `QueryKey`. Cache cleared on new archetype creation only.

## 9. Traceability to Decisions
- D-004: AOT-safe, no reflection in hot paths → reflection isolated to `ComponentTypeRegistry.GetOrRegister<T>()`, one-time per type, cached; M4 source-gen will eliminate it.
- D-023: Component storage design → chunked SoA archetype storage implemented per decision.
- D-042: Attribute-based storage routing (`[ArchetypeStorage]` / `[SparseStorage]`) → implemented.
- D-043: Default chunk size 16 KB; minimum capacity 1 → implemented in `ArchetypeRegistry.CreateArchetype`.
- D-044: `in T` / `out T` parameter model for query callbacks → `QueryCallback<T...>` delegates use `in` parameters.
- D-045: Permanent `QueryEach` API (not source-gen only) → implemented on `EcsWorld`.
- D-046: Lazy query cache with `PreloadQuery` for eager warm-up → implemented in `QueryRegistry`; `EcsWorld.PreloadQuery<T...>` exposed publicly.

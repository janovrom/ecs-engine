# ECS Engine Code Style Guide

## Naming Conventions

### Fields & Properties
- **Readonly fields**: PascalCase (e.g., `ReadonlyFieldName`)
- **Private fields**: _camelCase with underscore prefix (e.g., `_privateField`)
- **Properties**: PascalCase (e.g., `PropertyName`)
- **Constants**: UPPER_SNAKE_CASE (e.g., `CONSTANT_VALUE`)

### Classes & Types
- **Regular classes**: PascalCase (e.g., `MyClass`)
- **Abstract classes**: PascalCase ending with "Base" (e.g., `ComponentBase`, `SystemBase`)
- **Interfaces**: PascalCase starting with "I" (e.g., `IComponent`, `IEvent`)
- **Enums**: PascalCase (e.g., `EntityState`)

## Code Style

### Collections & Initialization
- Use collection initializers: `new List<T>() { }` or `[]` for target-typed expressions
- Prefer explicit types over `var`:
  ```csharp
  // Good
  List<Entity> entities = [];
  
  // Avoid
  var entities = new List<Entity>();
  ```
- Use simplified new for object creation:
  ```csharp
  // Good
  SomeClass instance = new();
  
  // Avoid
  SomeClass instance = new SomeClass();
  ```

### Namespaces
- Use file-scoped namespaces (no closing brace):
  ```csharp
  namespace EcsEngine.Core;
  
  public class MyClass { }
  ```

### Class Organization
Order class members as follows (no regions):
1. Data members (private fields)
2. Properties (including auto-properties)
3. Constructors
4. Override methods
5. Public methods
6. Private methods

Example:
```csharp
public sealed class MyClass
{
    private int _value;
    private List<string> _items = [];

    public int Value { get; set; }
    public IReadOnlyList<string> Items => _items;

    public MyClass(int initialValue)
    {
        _value = initialValue;
    }

    public override string ToString() => $"MyClass({_value})";

    public void AddItem(string item)
    {
        _items.Add(item);
    }

    private void ValidateItem(string item)
    {
        if (string.IsNullOrEmpty(item))
            throw new ArgumentException("Item cannot be empty.");
    }
}
```

### Class Sealing
- Seal classes when they are not intended to be subclassed:
  ```csharp
  public sealed class ConcreteClass { }
  ```

## Test Patterns

### Unit Tests
- Use NUnit with `[TestFixture]` and `[Test]` attributes
- Include Arrange/Act/Assert comments for clarity:
  ```csharp
  [Test]
  public void SomeMethod_WithCondition_ExpectedBehavior()
  {
      // Arrange
      var dependency = Substitute.For<IDependency>();
      var subject = new ClassUnderTest(dependency);

      // Act
      var result = subject.DoSomething();

      // Assert
      Assert.That(result, Is.EqualTo(expected));
  }
  ```

### Integration Tests
- Use NUnit with `[TestFixture]` and `[Test]` attributes
- Include Arrange/Act/Assert comments for clarity
- No mocking libraries (NSubstitute) in integration tests

## No Regions
- Do not use `#region` / `#endregion` directives
- Organize code by ordering class members appropriately instead

## AOT & Release Safety
- No reflection in release paths
- Constructor injection only for dependencies
- Avoid dynamic types and delegates where possible in hot paths

## Performance Guidelines
- Hot-path operations should target zero allocations
- Use value types for components (structs)
- Use collection initializers to avoid intermediate allocations
- Profile allocations in benchmarks for critical paths
